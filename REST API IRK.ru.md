Главный принцип REST API это наличие у каждого endpoint\* пяти основных url (часто будут работать только несколько, например для сущностей с правом доступа только для чтения будут работать только List, Get и Actions) и дополнительных опционально для выполнение особых сценариев (actions)

endpoint - url представляющий какую либо сущность

> например http://www.irk.ru/api/news/article/ - путь к сущности статьи в разделе новостей
# Основные методы endpoin

## List

По умолчанию если отправить GET запрос на endpoint то вернется список объектов в фаормате JSON (Обычно спагинированный и в ответе будет написана ссылка на следующую страницу если она есть)

list можно фильтровать, GET параметры для фильтрации нужно смотреть в документации

## Add

При отправке POST запроса на endpoint в теле которого будет JSON описание объекта (структуру нужно смотреть в документации) в ответе будет JSON объекта с сгенерированными полями (поля по умолчанию, id и прочее)

## Update

Если отправить PUT запрос на url в формате `endpoint/[id]/` в теле которого будет JSON обект, то объект обновится и в ответ прийдет JSON обновленного объекта

## Get

Если отправить GET запрос на url в формате `endpoint/[id]/`, то в ответ прийдет JSON объекта

## Delete

Если отправить DELETE запрос на url в формате `endpoint/[id]/`, то в ответ прийдет JSON со статусом успеха операции

## Actions

Особые методы API, которые помогают совершать определенные действия над объектом или в целом с сущностями. Например у события в афише может появиться метод `by_ticket`, который будет служить получением данных для покупки билета или совершать покупку. У такого метода будет url  `endpoint/[id]/by_ticket/`, т.к. он привязан к конкретному событию. Пример с общим методом, например мы можем совершить несколько действий над событиями и по завершении всех запросов хотим почистить кэш событий вызвав `clear_cache` url в таком случае будет выглядеть `endpoint/clear_cache/`
